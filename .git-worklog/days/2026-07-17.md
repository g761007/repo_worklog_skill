### 語言契約：BCP 47 解析器、config 讀取器與跨語言的摘要標記

- **異動內容：** 新增 `git_worklog/language.py`（BCP 47 語言標籤解析與依優先序決定輸出語言）、`git_worklog/config.py`（第一個真正讀取 config.json 的模組），並在 markers.py 為日期檔加上獨立的 SUMMARY 標記、為 index.md 加上語言標記，讓索引在被重建時語言只決定一次、之後保持不變；CLI 新增 `--interface-language`，`doctor`／`validate` 補上語言相關檢查。
- **程式碼行為：** `resolve` 的解析順序（明確要求 → CLI 參數 → 專案 config → 環境變數 → 系統 locale → 英文）中，agent-hosted 執行時 `allow_locale` 一律關閉，因為容器或遠端環境的作業系統 locale 說明不了使用者實際要求的語言。`config.py` 的 `load`／`language`／`index_language` 把 config.json 裡的 `"auto"` 一律視為「還沒有人設定過」而非一個選擇——因為 `ensure_data_dir` 從不覆寫既有檔案，每個既有工作日誌的 config.json 早就寫著 `auto`，這個值在磁碟上跟從沒被打開過是無法分辨的。索引摘要的抽取從原本寫死尋找 zh-TW「當日摘要」標題改成優先找 SUMMARY 標記（`summarise_generated`／`has_summary_marker`），避免非 zh-TW 語言寫的日期檔在重建索引時被誤判成沒有摘要而留白；沒有標記的舊檔仍退回標題比對，是永久的相容路徑而非過渡期。index.md 的 GENERATED 標記加上 `lang=` 屬性（`index_language_of`／`render_index`），讓索引語言在被第一次決定後保持穩定，不會因為某次重建換了語言的 agent 而每次翻轉。
- **實作方式：** `language.normalize`／`resolve`／`resolve_interface` 三個函式構成整個解析流程；`_TAG_RE` 把主要子標籤限制在 2–3 個字母（ISO 639-1/2/3 的實際長度），刻意不接受 4 個字母以上的子標籤，避免像「chinese」「garbage」這類拼字錯誤或垃圾 locale 被誤判成合法語言標籤。`markers.py` 新增 `_SUMMARY_MARKER_RE`／`render_summary`／`_between_summary_markers`／`index_chrome`／`index_language_of`。
- **影響範圍：** `build_analysis_manifest.py`／`collect_day_results.py`／`preview_state.py` 依序把解析後的語言帶進 manifest、寫進結果檔、記到 preview 指紋；SKILL.md 與 subagent-contract.md 同步告訴協調端與各個 subagent 該用哪個語言輸出。

### 把分析流程搬進套件，並新增 analyze prepare / collect 兩個子命令

- **異動內容：** 既有的 collect_git_history.py／build_analysis_manifest.py／collect_day_results.py 邏輯搬進 `git_worklog/analysis/` 底下（history.py／manifest.py／results.py），scripts/ 底下同名檔案改成轉呼叫套件；同時新增 `git-worklog analyze prepare` 與 `git-worklog analyze collect` 兩個 CLI 子命令，讓整個分析流程的「準備」與「回收驗證」兩端可以直接用 CLI 驅動，不必手動串接三支 script。
- **程式碼行為：** `prepare` 一次把整個日期範圍的每一天各自建立一份 Analysis Manifest 寫進 `tasks/<date>.json`，並回傳每個 subagent 該寫入的 `result_path`；`collect` 讀回 `results/<date>.json`，逐一對照 §6 schema 驗證後彙整成 `complete`／`degraded`／`missing`／`invalid` 等分類。兩個目錄分開（tasks/ 與 results/）讓 collect 可以分辨「manifest 發出去了但沒人回覆」跟「有一份結果但沒人要求過」兩種不同狀況。
- **實作方式：** `analysis/manifest.py` 的 `build` 組出完整 Analysis Manifest（changed_files、file_groups、required_context 等欄位）；`analysis/results.py` 的 `read_run` 逐一讀取每個日期的結果檔並呼叫 `validate` 檢查；`cli/analyze.py` 的 `_prepare`／`_collect` 是這兩端在 CLI 上的入口。
- **影響範圍：** README／CHANGELOG 記錄新指令；`tests/helpers.py` 新增或調整既有測試輔助函式以支援新的套件路徑。

### report 模式的統一入口，加上 ref 範圍與日期檔引用的核對機制

- **異動內容：** 新增 `git-worklog report`，一次決定產出報告前必須確認的四件事：範圍是日期還是 tag 的 commit 集合、這段範圍是否有分析可用（沿用 coverage）、對 tag 範圍而言哪些日期檔引用的 commit 其實屬於別的版本必須排除、以及輸出語言。同時新增 `analysis/reconcile.py`，把過去只寫在 references/report-mode.md 裡、要求模型自己遵守「只描述 commits[] 範圍內的工作」的一句話，變成真正可以機械檢查的資料。
- **程式碼行為：** `report` 本身不寫任何文字——「寫什麼」是 agent LLM 的工作，這個指令只決定「可以寫什麼」，所以刻意沒有 `--out` 參數。針對 tag 範圍，`reconcile` 掃描每個相關日期檔 GENERATED 區塊裡出現的 Git 物件名稱（不是解析中文的「相關 commits」標籤文字，因為那個標籤在非 zh-TW 語言寫的日期檔裡會被翻譯，照標籤找只會在其他語言的日期檔裡找到零筆引用卻誤判成乾淨），把每個引用分成四類：`backed`（範圍內且被引用）、`unbacked`（範圍內但沒有任何日期檔引用，可能是分析缺口或日期檔單純沒提到）、`out_of_range`（被引用但其實屬於別的版本，是報告必須排除的部分）、`unresolved`（引用了但解不出剛好一個 commit，通常是巧合像雜湊的英文詞或有歧義的短前綴）。一個前綴同時符合範圍內兩個以上 commit 時算誰都不算，寧可讓它落到待確認清單，也不能讓報告聲稱涵蓋了一個其實沒被分析過的 commit。
- **實作方式：** `reconcile.cited_hashes` 用正則式在 GENERATED 區塊掃描 7–40 碼的十六進位字串；`reconcile` 用一次 `git cat-file --batch-check` 批次解析所有候選字串對應的 commit，而非逐一呼叫，因為一份 90 天的報告可能要核對數百個引用；`cli/report.py` 的 `_run` 依序解析範圍（`refs.resolve` 或日期範圍）、呼叫 `coverage.check`，範圍是 ref 時再呼叫 `reconcile.reconcile`。
- **影響範圍：** SKILL.md／references/report-mode.md 同步記錄 report 指令的用法；`cli/__init__.py` 新增 report 子命令的參數定義。

### collect 除了核對 evidence[] 也核對文字段落裡引用的程式碼符號

- **異動內容：** 在既有的 evidence 對照樹驗證之外，新增對 `summary`／`implementation`／`behavior_change`／`impact`／`risks`／`maintenance_notes`／`follow_ups`／`handoff_notes`／`uncertainties` 等文字段落的檢查：任何用反引號標出來的識別字，都必須在當天的樹（起始或結束狀態）裡真的能找到，找不到就以 PROSE_SYMBOL_NOT_FOUND 判定整天失敗。這個工作補的是先前 evidence[] 驗證機制沒有涵蓋到的落差——`evidence[]` 被檢查過了，但 `implementation` 文字裡憑空捏造的名稱完全沒人核對過。
- **程式碼行為：** `cited_symbols` 只把「整段反引號內容都是一個識別字（可以有點分隔，可以有結尾的括號）」的片段視為符號，路徑、旗標、語言標籤、日期這類合法使用反引號的內容因為含有無法通過識別字判定的字元（斜線、連字號等）而被排除，維持這個檢查的雜訊夠低才有意義。`Tree.names_anything` 用 `git grep -qwFI` 對整個 repo 樹搜尋，而非只搜當天改動過的檔案，因為文字裡合理地會提到當天沒有改動過的既有程式碼；`day_trees` 回傳當天commit序列的起始（第一個commit的父提交）與結束兩棵樹，而非只看結束狀態，因為一個當天被刪除的符號在描述「刪除」這件事時仍然正確，只看結束狀態會把每一次刪除都誤判成捏造。
- **實作方式：** `analysis/results.py` 新增 `_prose_strings`（攤平字串或字串陣列欄位）、`cited_symbols`（用 `_CODE_SPAN_RE` 抓出反引號片段再篩出真正的識別字）、`validate_prose`；`Tree` 類別新增 `names_anything`／`day_trees` 兩個方法；`validate`／`read_run` 都新增 `commits` 參數，範圍刻意來自 manifest 而非結果檔自己回報的 `commits[]`——一個由被檢查對象自己決定範圍的檢查等於沒有檢查。
